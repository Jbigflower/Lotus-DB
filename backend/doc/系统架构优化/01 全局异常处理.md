
## 1️⃣ 架构概览

```
Client
  ↓
Router（API 层）
  ↓
Service（服务控制层）
  ↓
Logic（业务逻辑层）
  ↓
Repo（数据访问层）
  ↓
DB / Redis / 外部服务
```

> **目标**：让每一层各司其职，逻辑清晰、易于测试与扩展。

---

## 2️⃣ 各层职责说明

|层级|职责|输入 / 输出|示例文件|
|---|---|---|---|
|**Router**|处理 HTTP 请求、调用 Service 层、由 FastAPI 自动校验输入参数|Request → Response|`src/api/routers/movies_router.py`|
|**Service**|控制业务流向、是否异步执行（Celery）、异常与事务控制|调用 Logic，返回结果或任务 ID|`src/services/movies_service.py`|
|**Logic**|执行业务核心逻辑、封装算法与业务规则|输入模型 → 输出数据对象|`src/logic/movies/library_logic.py`|
|**Repo**|封装数据库访问（Mongo、Redis、SQLAlchemy）|输入查询参数 → 返回模型实例|`src/repos/mongo_repos/movies_repo.py`|
|**DB**|底层数据库驱动层（ODM/ORM/原生操作）|CRUD 执行层|`MongoEngine / Motor / SQLAlchemy`|

---

### 🧩 Router 层

**职责**：

- 接收 HTTP 请求并绑定至 Pydantic 模型
    
- 调用对应 Service
    
- 返回标准化响应（使用统一的响应模型）
    

**示例：**

```python
@router.post("/movies/", response_model=MovieRead)
async def create_movie(data: MovieCreate, service: MovieService = Depends()):
    return await service.create_movie(data)
```

---

### 🧩 Service 层

**职责**：

- 控制业务流程的执行方式
    
- 判断是否通过 Celery 异步执行
    
- 捕获并转译 Logic 层的异常
    
- 记录日志 / 指标
    

**示例：**

```python
class MovieService:
    def __init__(self):
        self.logic = MovieLogic()

    async def create_movie(self, data: MovieCreate):
        try:
            if data.background_task:
                # 异步执行
                task_id = send_to_celery("create_movie", data.model_dump())
                return {"task_id": task_id}
            else:
                # 直接执行逻辑层
                return await self.logic.create_movie(data)
        except Exception as e:
            logger.exception("[MovieService] create_movie failed")
            raise RepoError(str(e))
```

---

### 🧩 Logic 层

**职责**：

- 核心业务逻辑与领域规则
    
- 聚合多个 Repo 调用
    
- 保持数据一致性（逻辑层面）
    

**示例：**

```python
class MovieLogic:
    def __init__(self):
        self.movies_repo = MoviesRepo()
        self.assets_repo = AssetsRepo()

    async def create_movie(self, data: MovieCreate):
        movie = await self.movies_repo.insert_one(data)
        if data.assets:
            await self.assets_repo.link_assets(movie.id, data.assets)
        return movie
```

---

### 🧩 Repo 层

**职责**：

- 与数据库交互（MongoDB / Redis）
    
- 封装 CRUD、分页、查询等操作
    
- 转化数据库异常 → RepoError
    

**示例：**

```python
class MoviesRepo(BaseMongoRepo):
    collection_name = "movies"

    async def insert_one(self, data: MovieCreate):
        try:
            result = await self.collection.insert_one(data.model_dump())
            return await self.find_by_id(result.inserted_id)
        except Exception as e:
            raise RepoError(f"Failed to insert movie: {e}")
```

## 3️⃣ 异常与日志流转

### 异常流转链

```
Repo → Logic → Service → Router → Global Exception Handler
```

#### 各层异常处理原则：

| 层级          | 行为                                      |
| ----------- | --------------------------------------- |
| **Repo**    | 捕获底层异常 → 转换为 RepoError                  |
| **Logic**   | 抛出业务异常（如 ValidationError、NotFoundError） |
| **Service** | 捕获异常，记录上下文日志，再抛出                        |
| **Router**  | 不捕获，由全局 handler 统一处理                    |
| **Core**    | 全局 `AppBaseException` 捕获与格式化响应          |

#### 日志内容建议：

- 记录层名 + 操作名 + 关键 ID（如 user_id, movie_id）
    
- 异常统一使用结构化 JSON 输出
