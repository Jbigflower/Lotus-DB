🌲 一、整体思路

日志的核心目标是：

> “让每一层都只记录它能**理解的上下文**和 **能负责的异常** 。”

你可以理解为：

* **越上层，越贴近业务语义** （例如用户请求、参数、任务执行）；
* **越下层，越贴近技术细节** （例如数据库操作、缓存、网络交互）；
* 所有层的异常都要「就地记录 + 抛出包装后的异常」，不应该“吞掉”异常。

---

## 🧩 二、日志分层职责详解

### **1️⃣ Router 层（接口层）**

负责内容：**请求入口与响应出口的全链路记录**

| 日志目的 | 示例内容                               | 日志级别              |
| -------- | -------------------------------------- | --------------------- |
| 请求日志 | 请求路径、方法、参数、用户ID、trace_id | `INFO`              |
| 响应日志 | 状态码、执行时长、是否成功             | `INFO`              |
| 参数异常 | 请求校验失败、类型错误等               | `WARNING`           |
| 性能监控 | 每个 API 耗时统计                      | `DEBUG` or `INFO` |

✅** ****Router 层日志建议：**

* 使用 FastAPI 的中间件记录请求/响应。
* 保留 trace_id（请求链路ID），贯穿整个请求流程。
* 不记录敏感参数（如密码、token）。

---

### **2️⃣ Service 层（服务控制层）**

负责内容：**任务调度、异步执行、全局异常封装**

| 日志目的     | 示例内容                                 | 日志级别              |
| ------------ | ---------------------------------------- | --------------------- |
| 异步任务控制 | 是否启用 Celery、任务ID、调度状态        | `INFO`              |
| 外部服务调用 | 第三方API调用结果、状态码                | `INFO` or `ERROR` |
| 全局异常封装 | logic 或 repo 抛出的异常被统一处理       | `ERROR`             |
| 服务入口输出 | 哪个逻辑流被触发（例如“创建电影任务”） | `INFO`              |

✅** ****Service 层日志建议：**

* 记录逻辑的“开始”和“结束”。
* 如果启用 Celery，记录任务提交和执行状态。
* 封装底层异常后重新抛出业务异常（不要吞掉堆栈）。

---

### **3️⃣ Logic 层（业务逻辑层）**

负责内容：**核心业务流程与分支行为**

| 日志目的 | 示例内容                                 | 日志级别  |
| -------- | ---------------------------------------- | --------- |
| 业务流转 | 当前业务阶段（加载资源 → 校验 → 写库） | `INFO`  |
| 条件分支 | 特殊分支逻辑、重试、回退等               | `DEBUG` |
| 异常追踪 | Repo 层异常 + 自定义业务异常             | `ERROR` |
| 性能瓶颈 | 某些业务步骤耗时（如视频转码）           | `DEBUG` |

✅** ****Logic 层日志建议：**

* 这是系统“业务真相”的主战场。
* 记录逻辑决策和关键数据（但不要记录全量数据）。
* 如果遇到意料之外的异常，打日志 + 抛业务异常。

---

### **4️⃣ Repo 层（数据仓储层）**

负责内容：**数据库/缓存/文件IO 操作的追踪**

| 日志目的    | 示例内容                     | 日志级别  |
| ----------- | ---------------------------- | --------- |
| DB 查询日志 | 查询条件、返回数量、执行耗时 | `DEBUG` |
| DB 异常日志 | 连接错误、索引异常、写入失败 | `ERROR` |
| Redis 操作  | key、expire、命中率          | `DEBUG` |
| IO 操作     | 文件路径、存储大小           | `INFO`  |

✅** ****Repo 层日志建议：**

* BaseRepo 中应封装统一的日志接口。
* 在低层日志中 **避免业务信息** （比如用户ID、业务名称），只关注操作目标。
* 异常一律封装成** **`RepoError` 抛出给上层。

---

## ⚙️ 三、日志穿透机制（trace_id）

为了串联整个调用链（Router → Service → Logic → Repo），建议：

* 每个请求生成唯一的** **`trace_id`
* 通过** **`contextvars` 或** **`log_context` 在日志中自动注入
* 这样你就能用一个 trace_id 查看整个请求的所有日志片段

示例：

```
[2025-10-13 22:35:41] [INFO] [trace:7cf4e91b] User lotus 发起创建电影请求
[2025-10-13 22:35:41] [INFO] [trace:7cf4e91b] 启动后台 Celery 任务
[2025-10-13 22:35:41] [DEBUG] [trace:7cf4e91b] 执行 Mongo 写入库操作：movies.insert_one()
```

## 🧭 四、总结表格

| 层级    | 关注点     | 日志重点                     | 日志级别建议         |
| ------- | ---------- | ---------------------------- | -------------------- |
| Router  | 请求与响应 | trace_id、参数、状态码、耗时 | INFO / WARNING       |
| Service | 服务控制   | 任务执行、异常封装、外部接口 | INFO / ERROR         |
| Logic   | 核心逻辑   | 流程流转、条件分支、异常追踪 | INFO / DEBUG / ERROR |
| Repo    | 数据访问   | 数据操作、执行耗时、底层异常 | DEBUG / ERROR        |
