# Lotus-DB 项目基础与接口规范

本规范用于统一后端与前端的编码与接口约定，约束分层架构与数据模型的一致性，减少跨层耦合与重复逻辑。与《doc-llm/开发手册.md》配套使用，遇到冲突以本规范与设计文档为准。

---

## 一、项目基础规范

### 1. 代码风格（Python）

- 命名规则
  - 模块/文件：`snake_case.py`
  - 类：`PascalCase`
  - 函数/方法/变量：`snake_case`
  - 常量：`UPPER_CASE`
  - 私有成员：前缀 `_`，不导出到公共 API
- 导入顺序
  - 依次为：标准库 → 第三方库 → 本地项目（绝对导入），各组之间空行
  - 禁止 `from x import *`，优先显式导入；必要时使用别名保证可读性
- 类型与文档
  - 使用类型标注（PEP 484）；公共函数必须有简要 docstring（目的、参数、返回）
  - 避免过度复杂的类型体操，保持可读性
- 异常处理约定
  - 不吞异常；在最合适的层进行捕获与转换（详见异常规范）
  - 自定义领域异常基类：`DomainError`，分层子类：`RepoError` / `ServiceError`
  - 日志记录使用结构化格式，避免打印堆栈到用户响应

### 2. 代码风格（前端 Vue + TS）

- 组件与文件命名
  - 组件：`PascalCase`（如 `MovieCard.vue`）
  - 路由视图：`PascalCase`（如 `LibraryPage.vue`）
  - TypeScript 类型文件：`types/` 下 `PascalCase.ts` 或领域聚合 `index.ts`
  - 组合式函数：`useXxx.ts`（如 `useMovieService.ts`）
- 状态管理（Pinia）
  - 每个领域一个 store：`useMovieStore`、`useAssetStore`
  - Store 仅承载状态与轻量动作；网络调用委托 `services/` 层
  - 禁止在组件内直接调用 `axios`；统一走 `services/api.ts` 封装
- API 调用规范
  - 使用统一 `Axios` 实例：`baseURL=/api/v1`、超时、重试与错误拦截
  - 响应统一封装为 `{data, error, meta}`；失败时 `error` 不为 `null`
  - 统一在请求头携带 `X-Request-Id`（前端生成 UUID），便于日志追踪
- 其他约定
  - 优先使用 Composition API；Props/Emits 明确类型；避免在模板中书写复杂表达式
  - 样式遵循 BEM 或原子化方案；组件内样式作用域 `scoped`

### 3. 目录与模块划分

- 后端（建议）：
  - `src/db/`：连接与客户端封装（Mongo/Chroma/Redis）
  - `src/repos/`：数据访问封装（Mongo/Chroma/Redis）
  - `src/services/`：业务逻辑与流程编排
  - `src/models/`：Pydantic v2 模型（领域拆分）
  - `src/routers/`：FastAPI 路由（v1 按领域拆分）
  - `src/utils/`：通用工具（纯函数），禁止业务逻辑
- 前端（建议）：
  - `src/views/`、`src/components/`、`src/stores/`、`src/services/`、`src/types/`、`src/utils/`
  - `src/services/` 负责 API 封装与领域服务组合；`stores/` 仅保存状态与触发服务

### 4. 分层边界与禁止事项

- 必须在 Repo 层的逻辑
  - 与数据库/向量库/缓存的 CRUD、查询与聚合
  - ID、时间戳、索引相关的读写细节与转换（如 `_id` ↔ `id`）
  - 异常捕获与包装为 `RepoError`（不暴露底层驱动错误）
- 必须在 Service 层的逻辑
  - 跨仓储的组合、事务性策略与最终一致性处理
  - 缓存策略应用、搜索结果合并与重排、队列任务触发
  - 统一错误与返回结构的转换（供 Router 使用）
- 禁止跨层调用
  - Router 不得直接调用 Repo/DB
  - Service 不得直接使用 DB 客户端（仅通过 Repo）
  - Agents 仅调用 Service，不得越层操作 Repo/DB

### 5. 公共工具函数边界

- 可放入 `utils/` 的函数
  - 纯函数：字符串/集合/日期处理、分页与排序、校验与格式化
  - 通用外部服务小封装（如简单的签名/编码），不携带业务语义
  - 度量/监控辅助（如生成 `request_id`）
- 不得放入的内容
  - 任何依赖业务实体或存储结构的逻辑（应在 Service/Repo）
  - 读取配置并执行业务分支的逻辑（应在 Service）
  - 与数据库/网络的直接交互（应在 Repo/Service）
  - 未经测试的临时脚本与杂项

## 二、接口与数据规范

### 1. API 设计

- RESTful 路径
  - 资源使用复数：`/api/v1/movies`、`/api/v1/assets`
  - 子资源：`/api/v1/movies/{id}/assets`
  - 动作型端点需审慎：优先使用资源语义（如搜索：`/api/v1/movies/search`）
- 版本化策略
  - 路径前缀版本：`/api/v1`
  - 兼容修改（新增字段）不改版本；破坏性修改提升到 `v2`
  - 提供弃用标识与迁移说明
- 状态码约定
  - `200 OK`：成功读取或操作
  - `201 Created`：成功创建资源（Location 头可选）
  - `204 No Content`：成功但无返回体（如删除）
  - `400 Bad Request`：请求不合法（参数/格式错误）
  - `401 Unauthorized`：未认证
  - `403 Forbidden`：无权限
  - `404 Not Found`：资源不存在
  - `409 Conflict`：资源冲突（如重复导入）
  - `422 Unprocessable Entity`：数据校验失败（Pydantic 校验）
  - `500 Internal Server Error`：未预期错误（避免泄露细节）
- 统一响应格式
  - 成功：
    - `{ "data": any, "error": null, "meta": { "request_id": "uuid", ... } }`
  - 失败：
    - `{ "data": null, "error": { "code": "string", "message": "string", "details": {} }, "meta": { "request_id": "uuid" } }`
  - 分页 `meta`：`page`、`size`、`total`

### 2. 模型与 Schema

- Pydantic 模型来源与约束
  - 所有请求/响应模型必须在 `src/models/` 定义并复用，Router 不得临时定义新模型
  - 模型按领域拆分（movies/users/assets/...），命名以用途后缀：`Create`、`Update`、`Read`
  - 可通过 `Annotated`/`field validators` 保持数据一致性
- 字段统一策略
  - ID：由 Repo 层统一生成/转换（`_id` ↔ `id`），对外只暴露 `id`
  - 时间戳：`created_at`、`updated_at` 由 Repo 层维护，UTC 时区，ISO8601 字符串
  - 枚举/状态：集中常量定义并在模型中引用
  - 索引字段：在模型/文档中标识，确保 Repo 层建立索引

### 3. 异常与错误处理

- 捕获策略
  - Repo 层：捕获底层驱动异常（Mongo/Chroma/Redis），封装为 `RepoError` 并附最小上下文
  - Service 层：捕获 `RepoError`/业务异常，转换为统一错误码与消息；可触发补偿/重试
  - Router 层：统一将 `ServiceError` 映射为 HTTP 状态码与响应 envelope；未捕获异常交由中间件处理为 500
- 日志格式
  - 结构化日志字段：`timestamp`、`level`、`request_id`、`user_id?`、`route`、`error_code`、`message`、`context`
  - 避免输出内部堆栈与连接字符串；需要时记录到安全存储，不回传给前端
  - 关键操作写入 `logs` 集合（概要信息），文本日志用于细节排查

## 三、代码审查与 AI 辅助约束

- 架构一致性
  - 生成代码必须符合 `Router → Service → Repo → DB` 分层；不得出现未定义的层次/类名
  - 禁止组件越层调用；前端组件仅调用 `services/`，不得直接请求后端
- 审查流程
  - AI 生成的代码片段必须在合并前由人工审查与本地运行验证
  - 变更说明必须包含：影响范围、调用链、数据流、异常策略与测试覆盖点
  - 新增功能需附最小可行测试；涉及存储的改动需说明索引与迁移
- 风格与质量
  - 避免炫技与过度抽象；优先可读性与可维护性
  - 禁止将业务逻辑置于 `utils/`；禁止无类型的动态注入
  - 提交前通过格式与静态检查（建议：ruff/mypy/tsc/eslint）
