在个人媒体管理系统中，Redis 的典型用途包括：

1. **缓存层**
   * 热数据缓存（电影详情、用户资产列表、播放历史）
   * 查询结果缓存（高级检索、模糊搜索结果）
   * 提高读取性能，降低 MongoDB / FAISS 访问压力
2. **队列 / 后台任务管理**
   * 任务状态存储（导入任务、批量处理、向量化任务）
   * 消息队列（Pub/Sub 模式实现异步通知）
   * 与 FastAPI 后台任务或 Celery 等调度器配合
3. **短期状态或会话存储**
   * 用户临时偏好、搜索历史、筛选条件
   * API 请求限流、Token 过期管理
4. **计数器 / 排行统计**
   * 播放量、点击量、标签热度
   * 快速增减，无需写入 MongoDB

## 2️⃣ 各层职责分析

| 层级                               | Redis 职责                                                                                                                               | 开发者需要实现的功能                                                                                                                                                                                                       |
| ---------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Router（FastAPI 路由层）** | 只负责接收请求、返回响应。一般不直接访问 Redis，但可以触发缓存相关的 service 调用。                                                      | - 提供接口，例如 `/movies/{id}`、`/user/{id}/assets`。``- 可在文档或路由中标注哪些接口会触发缓存更新或读取。                                                                                                           |
| **Service（业务逻辑层）**    | Redis 核心业务逻辑封装层。``负责决定什么时候使用缓存、缓存策略、过期时间、缓存刷新逻辑。                                                 | - 查询时先从 Redis 读取，如果命中直接返回。``- 缓存未命中再访问 Repo（Mongo/lanceDB）并写入缓存。``- 实现缓存淘汰策略（LRU / TTL 等）和异步刷新。``- 处理临时会话数据、统计计数器。                                        |
| **Repo（仓储层）**           | Redis 只用于持久化轻量数据或快速访问的辅助存储；严格意义上数据源仍是 MongoDB/lanceDB。``Repo 可以封装 Redis 操作，但不包含复杂业务逻辑。 | - 提供基本 CRUD / get/set 接口：`get_movie_cache(movie_id)`、`set_movie_cache(movie_id, data, ttl)`。``- 提供批量操作接口，例如批量写入缓存、删除缓存。``- 封装计数器增减操作，如 `increment_play_count(movie_id)`。 |
| **DB（Redis 客户端封装层）** | 封装 Redis 连接、序列化/反序列化、连接池管理。                                                                                           | - 初始化 Redis 客户端（单例或连接池）。``- 提供统一接口给 Repo 使用，例如 `redis.get(key)`/`redis.set(key, value, ttl)`/`redis.hgetall(hash)`。``- 处理序列化（JSON、pickle 等）和异常重试。                               |

## 3️⃣ 开发者实现的功能清单

1. **DB 层（Redis 客户端封装）**
   * 初始化 Redis 连接池
   * 基本操作封装：get, set, hget, hset, incr, expire, delete
   * 异常处理、重连逻辑
   * 可选：支持 JSON 序列化/反序列化
2. **Repo 层**
   * 提供缓存接口给 Service
   * 计数器操作（播放量、热度等）
   * 可能的任务状态存储接口（任务进度、导入任务）
3. **Service 层**
   * 缓存策略：先读缓存 → 未命中再读 Repo → 写缓存
   * TTL 过期策略、缓存刷新
   * 复杂业务逻辑：统计排行榜、搜索结果缓存、批量缓存更新
4. **Router 层**
   * 调用 Service 接口
   * 可在接口文档中说明缓存命中率或 TTL 信息
   * 对外暴露统计接口或缓存状态接口（可选）

---


### 🔹 总结

* **DB 层** ：纯粹封装 Redis 客户端。
* **Repo 层** ：提供 Redis CRUD + 辅助接口。
* **Service 层** ：缓存策略和业务逻辑核心。
* **Router 层** ：只调用 Service，不直接操作 Redis。

> 核心原则： **Redis 只做快速访问/缓存/临时存储，不替代 MongoDB 的主数据存储** 。


## Lotus-DB Redis 缓存设计

| 模块                                                     | 是否需要缓存 | 缓存内容                                                               | 缓存用途                                         | TTL / 策略                                          |
| -------------------------------------------------------- | ------------ | ---------------------------------------------------------------------- | ------------------------------------------------ | --------------------------------------------------- |
| **Movie（核心电影资产）**                          | ✅           | 热门电影详情、搜索结果、卡片信息、推荐列表                             | 加速搜索和详情页展示，减少 MongoDB / Chroma 查询 | 1h 热门电影；搜索结果可 TTL 30min                   |
| **Assets（电影影片、字幕）**                       | ⚠️         | 元数据（大小、时长、格式、语言）、缩略图 URL、字幕文本向量（短期热用） | 快速获取影片信息、字幕检索                       | 元数据可 TTL 1h，字幕向量 LRU                       |
| **Library（媒体库配置）**                          | ✅           | 媒体库配置信息、目录结构、访问权限                                     | 页面快速加载库信息，切换库时无需频繁查询 Mongo   | TTL 12h 或按修改自动失效                            |
| **user（用户配置）**                               | ✅           | 登录会话信息（JWT）、偏好设置、权限信息                                | 会话验证、前端用户信息快速加载、权限检查         | JWT TTL 24h；权限/偏好 TTL 1h，修改即失效           |
| **watch_history（观看记录）**                      | ✅           | 最近 N 条观看记录、播放进度                                            | 播放页快速显示历史记录和继续播放                 | 最近 N 条列表可 TTL 1h，滚动淘汰                    |
| **user_asset（用户资产：截图、笔记、评论、剪辑）** | ⚠️         | 热门笔记、截图元数据、用户剪辑列表                                     | 快速展示用户资产列表，搜索笔记/截图              | 热门资产 TTL 30-60min，具体文件不缓存，只缓存元数据 |
| **user_custom_list（用户片单）**                   | ✅           | favorites、watchlist、自定义片单                                       | 快速渲染用户片单和播放列表                       | TTL 1h，修改即失效                                  |
| **task（任务相关）**                               | ✅           | 后台任务进度、状态、结果                                               | 任务中心实时显示进度、异步任务调度               | 实时更新，进度状态可短期缓存，完成后可 TTL 1h       |
| **向量化（内置 LRU）**                             | ✅           | embedding 向量缓存（电影描述、字幕、笔记）                             | 避免重复计算 embedding，快速检索                 | 内存限制 LRU 淘汰                                   |
| **celery 相关**                                    | ✅           | 异步队列数据、任务消息、调度信息                                       | Celery broker 支撑任务调度                       | 即时使用，消息消费后删除                            |
